## 第一部分：基础知识

### 1.1 声音是什么

当你说话时，声带在振动。这些振动推动空气，形成一圈圈看不见的"波"——就像往池塘里扔一块石头，水面会出现涟漪。这些声波传到别人的耳朵里，别人就听到了你的声音。

### 1.2 声音的数字化

计算机只能处理数字，所以需要把声波转换成数字。这个过程分两步：

**采样（Sampling）**——像高速拍照一样，每秒对声波"拍"几万张快照。常见的采样率：

| 采样率 | 含义 | 使用场景 |
|--------|------|---------|
| 48000 Hz | 每秒 48000 个数值 | LuxTTS 输出、专业音频 |
| 24000 Hz | 每秒 24000 个数值 | 大多数 TTS 引擎的标准输出 |
| 16000 Hz | 每秒 16000 个数值 | Whisper 语音转文字的输入 |

**量化（Quantization）**——每个采样点的振幅用一个数字表示（通常是 -1.0 到 +1.0 之间的浮点数）。

这样一段 10 秒的音频，在 24kHz 采样率下就是 240,000 个浮点数。

### 1.3 什么是声纹

每个人说话时，声带的粗细长短、嘴巴和鼻腔的形状都不一样，所以发出的声波也独一无二。声音也有"指纹"，叫做**声纹（Voiceprint）**。

声纹可以被提取成一串固定长度的数字向量（比如 256 维或 512 维的浮点数组），这串数字就代表了一个人的声音特征。

### 1.4 什么是声音克隆

声音克隆就是让计算机**学会一个人的声音特点**，然后用这个声音**说出任意的新话**。整个过程不需要那个人再开口说一个字，计算机就能用他的声音说出他从没说过的话。

打个比方：你有一个朋友能惟妙惟肖地模仿各种人的声音。你让他听一段班主任的声音，然后用班主任的声音念课文——声音克隆做的就是这件事，只不过做这件事的不是人，而是一个 AI 模型。

## 第二部分：核心技术原理

### 2.1 梅尔频谱图——声音的"热力图"

原始波形数据量太大，AI 很难直接处理。所以第一步是把波形转换成**梅尔频谱图（Mel Spectrogram）**。

你可以把它想象成一张"声音的热力图"：

- **横轴** = 时间（从左到右）
- **纵轴** = 频率（低音在下，高音在上）
- **颜色深浅** = 这个频率在这个时刻有多强

```
原始声波: ∿∿∿∿∿∿∿∿∿∿∿∿  (几万个数值，1-D 信号)
        ↓ 梅尔频谱转换（短时傅里叶变换 + 梅尔滤波器组）
频谱图:  ■□■□■■□□■■□■   (压缩后的 2-D 矩阵)
         时间 →
```

在 Voicebox 的 Chatterbox 引擎中，`log_mel_spectrogram` 函数（`backend/backends/base.py:309`）负责这个转换。梅尔频谱图之所以被广泛使用，是因为它模拟了人耳对频率的感知方式——人耳对低频变化敏感，对高频变化不敏感。

### 2.2 音频预处理——从原始录音到可用数据

在提取特征之前，原始音频需要经过一系列预处理。Voicebox 在 `backend/utils/audio.py` 中实现了完整的处理链：

#### 2.2.1 格式统一（重采样 + 单声道）

不同来源的音频格式各异（MP3、WAV、不同采样率、立体声/单声道），必须统一成标准格式：

```python
# 使用 librosa 加载音频，自动重采样到目标采样率，转成单声道
audio, sr = librosa.load(path, sr=24000, mono=True)
```

- **重采样**：librosa 自动将任意采样率的音频转换到 24kHz（或引擎需要的采样率）
- **单声道化**：立体声取平均值变成单声道，因为语音克隆不需要空间信息

#### 2.2.2 音量归一化（RMS Normalization）

不同录音的音量差异很大——手机录音可能很轻，专业麦克风录的又很响。归一化让所有音频处于同一音量水平：

```python
# 计算当前音量（RMS = 均方根）
rms = np.sqrt(np.mean(audio**2))

# 目标音量：-20 dB（相当于 0.1 的振幅）
target_rms = 10 ** (-20 / 20)   # ≈ 0.1

# 计算增益并应用
gain = target_rms / rms
audio = audio * gain

# 峰值限制：防止放大后超过 0.85（避免削波失真）
audio = np.clip(audio, -0.85, 0.85)
```

这个过程可以类比为：把所有录音调整到"同样大小的声音"，既不太大也不太小，方便 AI 模型处理。

#### 2.2.3 音频验证

上传的参考音频需要通过质量检查：

| 检查项 | 要求 | 原因 |
|--------|------|------|
| 时长 | 2 ~ 30 秒 | 太短没有足够信息，太长会有噪声干扰 |
| 音量（RMS） | ≥ 0.01 | 太安静或完全静音的音频无效 |
| 峰值 | ≤ 0.99 | 超过说明已经"削波失真"，质量不合格 |

#### 2.2.4 多样本拼接

如果用户上传了多个声音样本，需要合并成一个：

```
样本1: [音频数据1]  参考文本: "你好，我是小明"
样本2: [音频数据2]  参考文本: "今天天气不错"
样本3: [音频数据3]  参考文本: "很高兴认识你"
          ↓ 各自归一化后拼接 ↓
合并音频: [数据1][数据2][数据3]
合并文本: "你好，我是小明 今天天气不错 很高兴认识你"
```

Voicebox 用 MD5 哈希值标识样本组合，如果样本没变就直接用缓存的合并结果。

### 2.3 特征提取——AI 如何"听懂"声音

预处理完成后，就到了最关键的步骤：从音频中提取能让 AI 理解的数学特征。这是声音克隆的核心技术。下面从底层数学原理讲起。

#### 2.3.1 梅尔频谱图（Mel Spectrogram）——声音的"热力图"

原始波形是一维的振幅-时间序列，数据量大且难以直接反映人耳的感知特性。梅尔频谱图通过三个步骤将其转换成更适合 AI 处理的二维矩阵：

**步骤一：短时傅里叶变换（STFT）**

把整段音频切成小段（通常 20-50 毫秒一段），对每段做傅里叶变换，得到该段时间内的频率成分。

```
时间轴上的波形:
    ┌───┐   ┌───┐       ┌───┐
    │   │   │   │       │   │
────┘   └───┘   └───────┘   └────
    |←20ms→|←20ms→| ... |←20ms→|
        ↓       ↓             ↓
     傅里叶   傅里叶        傅里叶
     变换     变换          变换
        ↓       ↓             ↓
     频率1: 0.3  0.5  ...   0.2
     频率2: 0.1  0.8  ...   0.4
     频率3: 0.7  0.2  ...   0.6
     ...
```

傅里叶变换的直觉：就像把一杯混合果汁分解成"有多少苹果汁、多少橙汁、多少葡萄汁"——它把复杂的声波分解成不同频率的简单正弦波的叠加。

**步骤二：梅尔滤波器组（Mel Filter Bank）**

人耳对低频变化敏感，对高频变化不敏感。梅尔滤波器组模拟这一特性：
- 在低频区域放很多"窄"滤波器（精细分辨）
- 在高频区域放少量"宽"滤波器（粗略分辨）

通常使用 80 个梅尔滤波器，将数千个频率分量压缩成 80 个频带。

**步骤三：对数缩放**

人耳对音量的感知也是对数的（声音增强 10 倍，人耳只觉得"响了一点"）。所以对能量值取对数（dB）：

```python
log_mel = np.log(mel_energy + 1e-9)  # 加微小值防止 log(0)
```

**最终结果：**

```
梅尔频谱图 = 一个二维矩阵（80 × 时间帧数）
     ↓ 纵轴: 80 个梅尔频率通道
     ↓ 横轴: 时间帧
     ↓ 每个值: 该频率在该时刻的对数能量
```

在 Voicebox 的 Chatterbox 引擎中，`S3Tokenizer.log_mel_spectrogram` 函数就是做这个转换。代码中还有一个细节补丁（`patch_chatterbox_f32`）：因为 librosa 返回 float64 数据，但模型权重是 float32，需要在计算前做类型转换，否则会产生精度不匹配错误。

#### 2.3.2 X-Vector 与 Speaker Embedding——声纹向量的数学本质

梅尔频谱图描述了"声音在每一刻包含哪些频率"，但它还不是"声纹"。我们需要从中提取出**不随说话内容变化、只与说话人身份相关**的特征——这就是 Speaker Embedding（说话人嵌入）。

**X-Vector** 是目前最主流的声纹提取方法之一：

```
梅尔频谱图 (80 × T)
    ↓
[Frame-level 层] 多层 1D 卷积 + ReLU + BatchNorm
    ↓               逐帧提取局部声学特征
特征图 (512 × T')
    ↓
[Statistics Pooling] 计算均值和标准差
    ↓               把变长序列压缩成固定长度
统计向量 (1024)     = [均值512维, 标准差512维]
    ↓
[Segment-level 层] 全连接层
    ↓               提取说话人身份特征
X-Vector (512 维)   ← 这就是声纹向量
```

**为什么需要 Statistics Pooling？**

不同人说同样一句话，频谱图的时间长度可能不同（语速不同）。Statistics Pooling 通过计算整个时间维度上的**均值**和**标准差**，把任意长度的序列压缩成固定维度。均值捕捉"平均音色"，标准差捕捉"变化幅度"——两者合在一起就能描述一个人的声音特征。

在 Voicebox 的 Qwen3-TTS PyTorch 后端中，`create_voice_clone_prompt()` 函数有一个参数 `x_vector_only_mode=False`（`pytorch_backend.py:181`），说明 Qwen 不仅提取 X-Vector，还会附加更多上下文信息。

**声纹向量的直觉理解：**

你可以把 512 维的声纹向量想象成 512 个"声音属性评分"：

```
声纹向量 = [
    0.72,   ← 属性1: 声带粗细程度
    -0.15,  ← 属性2: 鼻腔共振强度
    0.33,   ← 属性3: 说话的基频
    0.89,   ← 属性4: 气流特征
    ...     ← （实际上这些维度是抽象的，没有明确的人类语言对应）
    -0.41   ← 属性512: 某种综合声学特征
]
```

虽然单个维度没有直观含义，但 512 个数值组合在一起就能精确描述一个人的声音。

**声纹向量的一个关键特性：余弦相似度**

两个人声音的相似度，通过计算两个声纹向量的**余弦相似度**来判断：

```
相似度 = cos(向量A, 向量B)
       = (A · B) / (|A| × |B|)

结果范围: -1 到 +1
  +1 = 完全相同的声音
   0 = 完全不相关
  -1 = 理论上的"完全相反"（实际中很少出现）
```

这就是为什么零样本克隆能工作的原因——模型能计算"新声音"和"已知声音"之间的相似度，从而在新声音的向量附近生成合理的语音。

#### 2.3.3 四种引擎的特征提取对比

Voicebox 中的 5 个支持克隆的引擎，各自使用了不同的特征提取策略：

**策略一：延迟处理（Chatterbox / Chatterbox Turbo）**

```python
# chatterbox_backend.py:131-142
async def create_voice_prompt(self, audio_path, reference_text, use_cache=True):
    voice_prompt = {
        "ref_audio": str(audio_path),   # 只存文件路径
        "ref_text": reference_text,     # 只存文本
    }
    return voice_prompt, False
```

Chatterbox 引擎在创建提示时**不做任何特征提取**——只记录文件路径。真正的声纹提取发生在调用 `model.generate()` 时，模型内部会：
1. 加载参考音频文件
2. 用 VoiceEncoder 提取梅尔频谱 → LSTM → 声纹向量
3. 用 T3 模型理解文字内容
4. 在 S3Gen 声码器中融合声纹和文字信息

VoiceEncoder 内部结构（从代码注释和补丁推断）：
```
参考音频 ──→ 梅尔频谱图 ──→ LSTM 神经网络 ──→ 声纹向量
                              ↑
                         多层 LSTM
                    捕捉时间序列中的
                    声纹特征模式
```

代码中的 `patch_chatterbox_f32` 函数（`base.py:293-328`）揭示了 VoiceEncoder 的两个关键入口：
- `S3Tokenizer.log_mel_spectrogram`：梅尔频谱计算
- `VoiceEncoder.forward`：LSTM 前向传播，接收梅尔频谱作为输入

**策略二：X-Vector + 上下文编码（Qwen3-TTS）**

```python
# pytorch_backend.py:175-191
def _create_prompt_sync():
    return self.model.create_voice_clone_prompt(
        ref_audio=str(audio_path),     # 参考音频路径
        ref_text=reference_text,        # 参考文本内容
        x_vector_only_mode=False,       # 不仅提取 X-Vector，还保留更多上下文
    )
```

Qwen3-TTS 使用 `create_voice_clone_prompt()` 一次性完成：
1. 加载音频 → 梅尔频谱
2. 提取 X-Vector（声纹向量）
3. 结合参考文本的音素对齐信息
4. 输出一个包含多维信息的提示字典

`x_vector_only_mode=False` 这个参数的含义是：除了声纹向量之外，还保留了参考音频和文本的更丰富的关联信息。这好比不仅记住了"这个人的声音特征"，还记住了"他读'你好'的时候是怎样的、读'世界'的时候又是怎样的"。

**缓存机制：** 提取结果会被缓存到内存和磁盘：

```python
# 缓存键 = 音频文件内容 + 参考文本 的 MD5 哈希
cache_key = hashlib.md5(audio_bytes + reference_text.encode("utf-8")).hexdigest()

# 先查内存缓存 → 再查磁盘缓存 → 都没有才重新计算
# 缓存文件格式: {cache_key}.prompt（PyTorch 序列化的字典）
```

这样同一份音频不需要每次都重新提取特征。

**策略三：自编码 + 自动转录（LuxTTS）**

```python
# luxtts_backend.py:126-133
def _encode_sync():
    return self.model.encode_prompt(
        prompt_audio=str(audio_path),   # 参考音频
        duration=5,                     # 取前 5 秒
        rms=0.01,                       # RMS 能量阈值
    )
```

LuxTTS 的特点：
- **不需要参考文本**：内部自动调用 Whisper ASR 做语音转文字
- **只取前 5 秒**：`duration=5` 限制了参考音频长度
- **输出一个紧凑的编码字典**：包含模型特定的压缩表示

它的 encode_prompt 内部流程是：
```
参考音频 ──→ [Whisper 自动转录] ──→ 参考文本
    ↓                                  ↓
    └──→ [音频编码器] ─────────────→ 音频特征编码
                                        ↓
                              合并 → 编码后的提示字典
```

**策略四：强制对齐编码（TADA）**

```python
# hume_backend.py:228-259
def _encode_sync():
    # 加载音频
    audio_np, sr = sf.read(str(audio_path), dtype="float32")

    # 强制对齐：音频中的每个词 → 对应的声音片段
    text_arg = [reference_text] if reference_text else None
    prompt = self.encoder(audio, text=text_arg, sample_rate=sr)

    # 将 EncoderOutput 序列化为字典（用于缓存）
    prompt_dict = {}
    for field_name in prompt.__dataclass_fields__:
        val = getattr(prompt, field_name)
        if isinstance(val, torch.Tensor):
            prompt_dict[field_name] = val.detach().cpu()
        ...
```

TADA 的编码器做的是**强制对齐（Forced Alignment）**：

```
参考文本: "你" "好" "世" "界"
参考音频: [音频波形数据......]
              ↓ 强制对齐 ↓
对齐结果:  [你: 0.0-0.3s] [好: 0.3-0.5s] [世: 0.5-0.8s] [界: 0.8-1.1s]
```

它精确地找到每个字在音频中的时间位置，然后为每个字提取对应的声音特征。这意味着模型不仅知道"这个人的声音听起来怎样"，还知道"每个字他是怎么发音的"。

如果用户没有提供参考文本，TADA 编码器内置了 ASR 功能（英语），可以自动转录。

#### 2.3.5 缓存机制——避免重复计算

特征提取是计算密集型操作（需要运行神经网络），所以 Voicebox 实现了两级缓存：

**一级缓存（内存）：**
```python
_memory_cache: dict[str, dict] = {}  # 进程内字典，速度最快
```

**二级缓存（磁盘）：**
```python
cache_file = cache_dir / f"{cache_key}.prompt"  # .prompt 文件，持久化
```

缓存键的计算方式：
```python
cache_key = MD5(音频文件的原始字节 + 参考文本的UTF-8编码)
```

这意味着：
- 相同的音频 + 相同的文本 → 命中缓存，瞬间返回
- 音频文件哪怕改了一个字节 → 缓存失效，重新提取
- 参考文本改了一个字 → 同样缓存失效

### 2.4 语音合成——从文字到声音

拿到声纹向量后，AI 模型要做的任务是：

```
输入：要说的文字 + 目标声纹特征 + 语言
输出：对应的音频波形
```

Voicebox 中不同引擎使用了三种不同的合成架构：

#### 架构 A：Transformer 编解码（Chatterbox 系列）

Chatterbox 内部由三个子模型协同工作：

```
文字 ──→ [T3 文本到 Token 模型] ──→ Token 序列
                                           ↓
参考音频 ──→ [VoiceEncoder] ──→ 声纹 ──→ [S3Gen 声码器] ──→ 音频波形
```

- **T3**（Text-to-Token）：理解文字内容，转换成中间的"声音标记"序列
- **VoiceEncoder**：从参考音频中提取声纹
- **S3Gen**（Speech Synthesis Generator）：把 Token 序列 + 声纹信息合成最终的声波

#### 架构 B：大语言模型（Qwen3-TTS）

Qwen3-TTS 基于 Transformer 大语言模型（参数量达 17 亿），把语音生成当成一种"翻译"任务——从"文字语言"翻译到"声音语言"：

```
文字 ──→ [Tokenizer 分词] ──→ [LLM 生成声音 Token] ──→ [Vocoder 声码器] ──→ 波形
              ↑                        ↑
          语言信息                  声纹提示
```

#### 架构 C：扩散模型（LuxTTS）

扩散模型的工作原理像"雕刻"：

```
随机噪声 ──────────→ [去噪步骤 1] ──→ [去噪步骤 2] ──→ ... ──→ 清晰音频
                        ↑                  ↑
文字特征 ─────────→ 条件引导        条件引导
声纹提示 ─────────→ 条件引导        条件引导
```

1. 从纯随机噪声开始（像一块粗糙的石头）
2. 在声纹和文字信息的"指导"下，一步一步去掉噪声
3. 每去一步，声音就清晰一点
4. LuxTTS 只需 4 步就能完成（速度极快）

#### 架构 D：因果语言模型 + Flow Matching（TADA）

TADA 基于 Llama 3.2（Meta 的大语言模型），是所有引擎中架构最独特的：

```
编码后的声音提示 + 目标文本 ──→ [Llama 3.2 因果LM] ──→ [Flow Matching 扩散] ──→ 音频
```

因为它底层是一个语言模型，能理解很长的上下文，所以可以生成 700 秒以上的超长连贯语音。

### 2.5 零样本克隆——为什么几秒就够

传统声音克隆需要：
- 让目标人物录制大量语音（几十分钟到几小时）
- 用这些数据专门训练一个模型（几小时到几天）

**零样本（Zero-shot）克隆**只需要：
- 几秒钟的参考音频
- 不需要额外训练
- 即刻就能克隆

这之所以可能，是因为模型在出厂前已经在海量语音数据上训练过了（通常几十万小时）。它已经学会了"什么样的声纹特征对应什么样的声音"，所以听到一小段新声音就能"理解"并模仿。

可以类比为：一个经验丰富的配音演员，听了别人说几句话，就能模仿那个人的声音——因为他已经听了无数人的声音，积累了丰富的"声音经验"。

### 2.6 长文本智能分句与拼接

对于长文本，Voicebox 有一个精心设计的分句算法。分句优先级：

```
1. 句末标点（。 . ! ? ）     ← 最佳切分点
2. 从句标点（; : , —）      ← 次选
3. 空格                     ← 兜底
4. 硬切（但不切开 [laugh] 等标签）
```

它还会：
- 跳过缩写词中的句号（如 "Dr."、"Mr."、"U.S."）
- 识别中文标点（。！？）
- 确保不会在 `[laugh]` 这样的情感标签中间切开
- 支持自定义分句长度（100-5000 字符）和交叉淡入淡出时长（0-200ms）

各段音频生成后，用**交叉淡入淡出（Crossfade）**拼接：

```
音频1结尾: ▇▇▇▇▅▃▂    ← 音量渐弱（fade out）
音频2开头:     ▂▃▅▇▇▇▇  ← 音量渐强（fade in）
拼接结果: ▇▇▇▇▅▅▅▇▇▇▇  ← 平滑过渡，没有"咔嗒"声
```

### 2.7 后期音效处理

生成语音后，Voicebox 使用 Spotify 开源的 Pedalboard 库对音频做后期加工，就像手机拍照后可以加滤镜一样。

**8 种音效：**

| 音效 | 作用 | 通俗解释 |
|------|------|---------|
| 变调（Pitch Shift） | 把声音变高或变低 | 变声器效果，最高升降 12 个半音 |
| 混响（Reverb） | 模拟不同空间的声音 | 让声音听起来像在教室/体育馆/山谷中 |
| 延迟/回声（Delay） | 添加回声效果 | 像对着山谷喊话："你好——你好——你好" |
| 合唱/镶边（Chorus） | 让声音更有厚度 | 一个人说话听起来像几个人同时说 |
| 压缩（Compressor） | 让音量更均匀 | 大声变小一点，小声变大一点 |
| 增益（Gain） | 调整音量大小 | 整体音量变大或变小（-40 到 +40 dB） |
| 高通滤波（High-Pass） | 去掉低频声音 | 去除"嗡嗡"的杂音 |
| 低通滤波（Low-Pass） | 去掉高频声音 | 让声音听起来像老式电话 |

**4 种内置预设：**

| 预设名 | 效果描述 |
|--------|---------|
| 机器人（Robotic） | 金属感的机械声音，通过镶边效果实现 |
| 电台（Radio） | 像老式 AM 收音机传出的声音，通过带通滤波 + 压缩实现 |
| 回声室（Echo Chamber） | 充满空间感的回声效果，通过大混响 + 延迟实现 |
| 深沉嗓音（Deep Voice） | 低沉有力的声音，通过降调 + 低通滤波 + 压缩实现 |

音效链的执行方式就像吉他效果器——音频信号依次通过每个"踏板"，每个踏板对声音做一次变换。

## 第三部分：Voicebox 工作流程

### 3.1 创建声音档案

用户通过录音或上传音频文件提供参考声音。系统会：

1. 验证音频格式和有效性
2. 将音频转换为标准格式（WAV，24kHz，单声道）
3. 对音量做归一化处理
4. 如果上传了多个样本，自动拼接成一段长音频
5. 保存到本地磁盘，创建声音档案记录

每个声音档案包含：唯一 ID、名称、描述、语言标签、头像，以及关联的音频样本。

### 3.2 加载 AI 模型

模型文件存储在本地 HuggingFace 缓存中。首次使用时从网上下载，之后直接加载。

加载过程会：
1. 检查模型是否已缓存到本地
2. 如果未缓存，从 HuggingFace Hub 下载（带进度追踪）
3. 将模型加载到内存或 GPU 显存
4. 根据 GPU 类型自动选择精度（CUDA/XPU 用 bf16，CPU 用 fp32）

### 3.3 生成语音

用户输入文字后，系统：

1. 为选定的声音档案创建声纹提示（从音频样本中提取）
2. 如果文本过长，自动分句
3. 对每段文字调用 AI 模型生成音频
4. 将各段音频用交叉淡入淡出拼接
5. 对最终音频做归一化处理

### 3.4 后期处理

如果用户配置了音效链：

1. 先保存一份纯净的"原始版本"
2. 按顺序应用音效链中的每个效果
3. 保存一份"特效版本"

### 3.5 保存和管理

- 音频保存为 WAV 文件
- 每次生成都会记录版本历史（原始版本、特效版本、多次尝试）
- 用户可以标记收藏、重新生成、重试失败的生成任务

| 项目 | 详情 |

### 4.3 LuxTTS

| 项目 | 详情 |
|------|------|
| 开发者 | ZipVoice 团队 |
| 模型体积 | ≈ 300 MB（最小的支持克隆的引擎） |
| 支持语言 | 仅英语 |
| 输出采样率 | **48000 Hz**（所有引擎中最高） |
| 克隆方式 | 零样本克隆——5 秒参考音频 |
| 显存需求 | ≈ 1 GB VRAM |
| HuggingFace 模型库 | `YatharthS/LuxTTS` |

**架构原理：** 基于扩散模型。生成过程从纯随机噪声开始，逐步"雕刻"出清晰语音。仅需 4 步去噪，速度极快——在 CPU 上达到 150 倍实时速度（生成 1 秒音频只需 0.007 秒）。

关键参数：
- `num_steps=4`：去噪步数（速度与质量的平衡）
- `guidance_scale=3.0`：声纹引导强度
- `t_shift=0.5`：时间偏移参数

声纹编码使用 `encode_prompt()` 函数，内部自动调用 Whisper 做语音转文字，所以用户不需要手动输入参考文本。

**适合场景：** 硬件配置较低（没有独显、内存有限），只需要英语语音，追求速度。

**适合场景：** 需要小语种语音克隆（阿拉伯语、芬兰语、斯瓦希里语等），或者需要最多语言覆盖时。

| 输出采样率 | 24000 Hz |
| 克隆方式 | 零样本克隆 |
| 独特功能 | 副语言标签（Paralinguistic Tags） |
| HuggingFace 模型库 | `ResembleAI/chatterbox-turbo` |

**副语言标签——让 AI 表达情绪：**

在文本中插入特殊标签，AI 会在说话时"表演"出这些声音：

| 标签 | 效果 |
|------|------|
| `[laugh]` | 笑声 |
| `[chuckle]` | 轻笑 |
| `[gasp]` | 倒吸一口气 |
| `[cough]` | 咳嗽 |
| `[sigh]` | 叹气 |
| `[groan]` | 呻吟 |
| `[sniff]` | 吸鼻子 |
| `[shush]` | 嘘声 |
| `[clear throat]` | 清嗓子 |

用法举例：`Hello, [laugh] it's so nice to meet you!`——AI 会在说话的同时笑一声。

**架构原理：** 和 Chatterbox 多语言版同源，但体积小得多（350M），使用 MeanFlow 声码器，速度更快。生成参数使用 `top_k=1000`、`top_p=0.95` 采样策略。

**适合场景：** 英语内容创作，需要让语音带有情绪表达（笑声、叹气等），比如播客、有声书、游戏配音。

### 4.6 TADA

| 项目 | 详情 |
|------|------|
| 开发者 | Hume AI |
| 参数量 | 1B（英语版）或 3B（多语言版） |
| 模型体积 | 1B ≈ 4 GB，3B ≈ 8 GB |
| 支持语言 | 1B：英语 / 3B：英、中、阿、德、西、法、意、日、波兰、葡 |
| 输出采样率 | 24000 Hz |
| 克隆方式 | 零样本克隆 |
| 独特功能 | 能生成 700 秒以上连贯语音 |
| HuggingFace 模型库 | `HumeAI/tada-1b`、`HumeAI/tada-3b-ml` |

**架构原理——文字-声学双重对齐（Text-Acoustic Dual Alignment）：**

TADA 基于 Llama 3.2（Meta 的大语言模型）构建，是所有引擎中架构最独特的。

```
┌──────────────────────────────────────────────────────────┐
│                     TADA 架构                              │
│                                                            │
│  [Encoder 编码器]                                          │
│    参考音频 + 文本 → 强制对齐 → 1:1 Token-音频对应关系      │
│    （每个字是怎么发音的，都记录下来）                         │
│                                                            │
│  [TadaForCausalLM 因果语言模型]                             │
│    基于 Llama 3.2 架构                                     │
│    接收编码后的声音提示 + 目标文本                            │
│    通过 Flow-Matching 扩散生成音频                          │
│                                                            │
│  [Codec 编解码器]                                           │
│    独立的音频编解码模块                                      │
│    将模型输出转换为最终波形                                  │
└──────────────────────────────────────────────────────────┘
```

核心创新是编码器会把参考音频中的每个词和对应的声音片段做精确的 1:1 对齐。这意味着模型不仅知道"这个人的声音听起来怎样"，还知道"每个字他是怎么发音的"。

因为底层是 Llama 语言模型，能理解很长的上下文，所以能生成超长（700 秒+）且语义连贯的语音。

**适合场景：** 需要生成很长的语音（如有声书章节、长篇播客），需要最高质量且硬件配置较好（8GB+ 显存）。

### 4.8 引擎对比总结

| 引擎 | 克隆 | 体积 | 语言数 | 特色 | 最佳用途 |
|------|:----:|-----:|:------:|------|---------|
| **Qwen3-TTS** | 支持 | 1.2-3.5 GB | 10 | 中文质量最高 | 中文语音克隆 |
| **Qwen CustomVoice** | 不支持 | 1.2-3.5 GB | 10 | 语气指令控制 | 用指令控制说话风格 |
| **LuxTTS** | 支持 | 300 MB | 1 | 48kHz、CPU 超快 | 低配置英语场景 |
| **Chatterbox** | 支持 | 3.2 GB | **23** | 语言最多 | 小语种克隆 |
| **Chatterbox Turbo** | 支持 | 1.5 GB | 1 | 情绪标签 | 英语播客/配音 |
| **TADA** | 支持 | 4-8 GB | 10 | 超长连贯语音 | 有声书、长篇朗读 |
| **Kokoro** | 不支持 | 350 MB | 9 | 54 种预置声音 | 开箱即用、CPU 友好 |

**选型建议：**

- 中文克隆 → Qwen3-TTS
- 小语种克隆 → Chatterbox
- 需要情绪表达 → Chatterbox Turbo
- 低配置/英语 → LuxTTS
- 超长文本 → TADA
- 不需要克隆、开箱即用 → Kokoro 或 Qwen CustomVoice

## 第五部分：配套模型——Whisper 语音转文字

Voicebox 还集成了 OpenAI 的 Whisper 模型，用于将录音自动转成文字（语音转文字 / STT）。这在创建声音档案时非常有用——你不需要手动输入参考音频的文字内容。

**5 种模型大小：**

| 模型 | 大小 | 速度 | 精度 |
|------|------|------|------|
| Whisper Base | ~74M 参数 | 最快 | 基础 |
| Whisper Small | ~244M 参数 | 快 | 良好 |
| Whisper Medium | ~769M 参数 | 中等 | 很好 |
| Whisper Large v3 | ~1550M 参数 | 慢 | 最佳 |
| Whisper Turbo | ~809M 参数 | 较快 | 接近 Large |

和 TTS 引擎一样，Whisper 在 macOS 上使用 MLX 加速，其他平台使用 PyTorch。

## 第七部分：伦理与安全

声音克隆是一项强大的技术，必须负责任地使用。

**应该做的：**
- 克隆自己的声音用于个人项目
- 在获得本人明确同意后克隆他人的声音
- 用于辅助功能（如帮助失去声音的人重新"说话"）
- 用于创意项目（动画配音、游戏制作、播客等）

**不应该做的：**
- 未经允许克隆他人的声音
- 用克隆的声音冒充他人进行诈骗
- 制造虚假的音视频来欺骗他人
- 在未告知听众的情况下使用克隆声音发布内容

Voicebox 的设计原则是 **本地优先（Local-first）**——所有模型和数据都保存在你自己的电脑上，不会上传到任何服务器。这最大程度地保护了隐私。
